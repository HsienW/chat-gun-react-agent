# 歷史包袱通用處理流程

## 使用時機

當需要處理下列任一情境時使用本 Skill：

- 既有模組需要泛化或重構（如 Weather 從硬編碼泛化為 Provider 驅動）。
- 發現技術債需要系統性清理。
- 模組職責模糊需要拆分或合併。
- 既有契約需要向後相容地遷移。
- 需要從「能運作」升級到「可維護、可測試、可擴展」。

## 強制前置條件

依序讀取：

1. 受模組的完整原始碼（含測試）。
2. 受影響套件的 `AGENTS.md`。
3. 相關能力域規則。
4. 既有 OpenSpec（如有）。
5. Git 歷史（`git log`、`git blame`）了解演進脈絡。

## Phase 1: 代碼分析

### 1.1 結構盤點

列出目標模組的：

- 所有公開函式、型別、常數。
- 檔案間依賴關係。
- 外部依賴（第三方套件、API、Provider）。
- 測試覆蓋範圍。

### 1.2 問題標記

針對每個檔案標記：

| 問題類型 | 說明 | 範例 |
| --- | --- | --- |
| 硬編碼 | 固定值散落在邏輯中 | 固定城市 allowlist |
| 職責混合 | 一個函式做多件事 | 意圖判斷 + API 呼叫 + 格式化 |
| 型別不安全 | `any`、type assertion | `as unknown as` 掩蓋不一致 |
| 契約缺失 | 無 Schema 驗證 | Tool Output 直接 trust |
| 測試缺口 | 未覆蓋的分支 | 只有 happy path |
| 耦合過緊 | 直接依賴具體實作 | 硬編碼 Provider URL |

### 1.3 影響範圍

繪製依賴圖：

```text
直接呼叫者 → 目標模組 → 直接依賴
     ↑                        ↓
  間接消費者              間接依賴
```

## Phase 2: 結構模塊 Research

### 2.1 識別可獨立重構的模塊

將目標模組拆分為可獨立處理的模塊：

- **輸入處理模塊**：正規化、驗證、意圖抽取。
- **核心邏輯模塊**：業務規則、狀態轉換。
- **外部整合模塊**：Provider、API、MCP。
- **輸出處理模塊**：格式化、序列化、Schema 驗證。
- **展示模塊**：UI 渲染、狀態映射。

### 2.2 評估每個模塊的重構難度

| 評估維度 | 低難度 | 高難度 |
| --- | --- | --- |
| 測試覆蓋 | 有完整測試 | 無測試或只有 snapshot |
| 耦合度 | 介面清晰 | 四處散落引用 |
| 契約穩定性 | Schema 已定義 | 型別靠猜 |
| 消費者數量 | 1-2 個 | 跨多套件 |

## Phase 3: 依賴分析

### 3.1 向內依賴（誰依賴我）

```bash
rg "from.*<module>" --glob "*.ts" --glob "*.tsx"
```

記錄所有消費者，標記：

- 是否依賴具體型別還是介面。
- 是否可以接受向後相容變更。
- 是否需要同步更新。

### 3.2 向外依賴（我依賴誰）

記錄所有外部依賴：

- 第三方套件版本。
- 外部 API endpoint。
- 環境變數或設定。
- 其他內部模組。

### 3.3 依賴方向檢查

確認依賴方向正確：

- 高層模組不得依賴低層具體實作。
- Provider 必須透過介面隔離。
- Domain 型別不得被 Transport 型別污染。

## Phase 4: 方案挑選

### 4.1 候選方案

針對每個重構目標，提出至少 2 個方案：

```text
方案 A: [名稱]
- 做法：
- 優點：
- 缺點：
- 破壞性：
- 工作量：

方案 B: [名稱]
- 做法：
- 優點：
- 缺點：
- 破壞性：
- 工作量：
```

### 4.2 決策矩陣

| 準則 | 權重 | 方案 A | 方案 B |
| --- | --- | --- | --- |
| 向後相容 | 高 | | |
| 測試可驗證性 | 高 | | |
| 實作複雜度 | 中 | | |
| 長期可維護性 | 中 | | |
| 跨層影響 | 高 | | |

### 4.3 選擇原則

1. 優先選擇向後相容的方案。
2. 優先選擇有明確驗證方式的方案。
3. 避免「順便重構其他東西」。
4. 單一變更只做一件事。

## Phase 5: 改造實作

### 5.1 執行順序

1. 先新增介面 / 型別（不破壞既有）。
2. 先寫測試（證明新行為正確）。
3. 逐步替換實作（保持每次可建置）。
4. 最後移除舊程式碼（確認無消費者後）。

### 5.2 向後相容策略

- 新舊型別共存期間，使用 `schemaVersion` 區分。
- 新增欄位使用 optional，不改變既有 required。
- 錯誤碼不得改變既有語意。
- 既有 Graph ID、Route、Event Type 保持相容。

### 5.3 停止線

符合以下任一條件時停止局部微調：

- 已進行兩輪修正仍未解決。
- 修復 A 後造成 B 回歸。
- 只能靠增加更多特殊分支維持。
- Mock 通過但真實 Runtime 仍失敗。

停止後必須：

1. 固化所有已知失敗案例。
2. 比較修改前後的輸入與輸出。
3. 判定真正失敗層級。
4. 重新決定修改目標。

## Phase 6: 驗收

### 6.1 驗證清單

- [ ] 既有測試全部通過（無刪除或放寬）。
- [ ] 新增測試覆蓋新行為。
- [ ] 跨層契約同步更新。
- [ ] Build 通過（無 type error）。
- [ ] Lint 通過。
- [ ] 手動驗收矩陣（Golden Cases）通過。
- [ ] 無硬編碼新增。
- [ ] Git Diff 不含無關修改。

### 6.2 回歸矩陣

針對重構目標建立回歸矩陣：

| 輸入案例 | 修改前結果 | 修改後結果 | 是否符合預期 |
| --- | --- | --- | --- |

## 參考案例：Weather 泛化

以 `generalize-weather-location-resolution` Change 為例：

- **Phase 1**：盤點 `weather.ts`（691 行）中的硬編碼城市、直接 API 呼叫、混合職責。
- **Phase 2**：拆分為 normalizer → resolver → provider → tool 四個模塊。
- **Phase 3**：確認 frontend `WeatherToolResult.tsx`、backend `deep-researcher.ts` 的依賴。
- **Phase 4**：選擇 Provider 驅動 + 結構化結果的方案。
- **Phase 5**：先建立 `weather-types.ts` 型別，再實作 `location-resolver.ts`，最後更新 tool。
- **Phase 6**：Golden Regression Matrix 驗證多語言地點、失敗案例、邊界輸入。
